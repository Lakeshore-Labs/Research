# Automated Engineering Manager: Upstream Judgment, Not Just Scheduling

**One-liner:** Build a manager agent that turns vague, unstructured backlog items into well-formed, well-assigned work (the upstream managerial judgment that's the real edge) and test whether that judgment beats rule-based coordination by checking the agent's calls against both a heuristic scheduler and *real human-manager decisions*.

**Area:** Research / SW   •   **Difficulty:** Advanced (the validity of the test is the hard, interesting part)

## Motivation
Coding agents make individual engineers much faster, which pushes the bottleneck up a level: from writing code to deciding who does what, in what order, and whether it's on track. The instinct is to frame this as an automation question: can an agent groom the backlog, assign work, track status, report up? But most of that is the *easy* part. Scheduling clean, well-specified tickets across a team is a solvable optimization problem; a decent heuristic (priority-first + skill-match + least-loaded) already wins there, and an agent has little to add.

The real edge of a good engineering manager is upstream of scheduling: taking a vague, unstructured backlog item (a free-text epic, a bug with no acceptance criteria, a dependency stated only in prose) and turning it into well-formed, well-assigned work. That's judgment, not optimization. So the capability worth building is an agent that does this coordination, and the generalizable insight worth chasing is **whether, and where, LLM judgment actually beats rule-based coordination**, versus where it just dresses up a scheduler a heuristic would match.

That insight only means something if the test is valid. The natural testbed is a simulated team, but a simulator carries a real risk: it can be circular, rewarding exactly the decisions its own designer baked in as "good," so the agent looks smart by agreeing with the sim's assumptions. The anchor against that is comparing the agent's calls to how a *real human manager* would have handled the same backlog. That human-decision comparison, not a metric dashboard, is how this project earns its conclusion. The simulator is the testbed; the human comparison is the validity check. The agent is scoped to coordination, because a chunk of management is irreducibly human and worth keeping out.

## Research Question(s)
1. **The capability:** can an LLM agent do the upstream managerial judgment, turning vague, unstructured backlog items (free-text epics, missing acceptance criteria, prose-only dependencies) into well-formed, well-assigned tickets, then keep the team coordinated (prioritize, track, unblock, report) acting only through tools (issue tracker, git/PRs, chat)?
2. **The insight:** where does LLM judgment actually beat rule-based coordination, and where does it merely match a heuristic? On clean backlogs a scheduler should tie the agent; the agent's plausible win is upstream, on the vague items a scheduler can't parse. Does that win show up as real downstream gains (less rework, more on-time work)?
3. **The validity check:** do the agent's calls line up with what a real human manager would do on the same backlog, and where they diverge, who's right? This is the guard against the simulator just rewarding its own assumptions.
4. **Failure and gaming:** does it propagate errors, overload its best worker, optimize a measured metric at the expense of an unmeasured one (Goodhart), or make confidently wrong calls that need a human override?

## Scope
- **In scope:**
  - A manager-agent loop that, each tick, reads team + backlog + board state and emits coordination actions: decompose a vague item into tickets with acceptance criteria; estimate and prioritize; assign and load-balance; flag blockers, dependencies, and risks; produce a status report.
  - A tool/interface layer the agent acts through, modeled on a real issue tracker (Jira / GitHub Issues), git/PRs, and a chat channel. Abstract it behind a small tool API so the same agent could later be pointed at a real tracker.
  - A simulated team + backlog as the testbed for development and testing. Workers parameterized by speed/skill/reliability consume assigned tickets, take stochastic time, sometimes block, sometimes return buggy work (rework), and emit progress updates. Crucially, the backlog includes a meaningful fraction of deliberately under-specified items so the upstream-judgment capability has something to chew on. This makes the system runnable and inspectable without a real org.
  - **A human-decision comparison, the central validity check.** Collect a small logged dataset of how a real human manager (a TA, a mentor, the student) would decompose, prioritize, and assign the same backlogs, and compare the agent's calls against it. This is what keeps the conclusion from being circular, i.e. the agent looking good only because it agrees with the simulator's built-in notion of "good."
  - An evaluation view: a comparison against baselines and the human decisions, plus a handful of metrics (throughput, cycle/lead time, on-time completion, rework/defect rate, load fairness, decomposition quality) and a check for metric-gaming. The metrics support the comparison; they are not the headline.
- **Out of scope (the human parts):**
  - Mentoring, coaching, career growth, performance review of real people, hiring/firing, morale, conflict resolution, compensation. The agent manages the work, not the people; that boundary is a deliberate design choice, not an afterthought.
  - Writing the feature code. The workers do that. This is orchestration, not a coding agent.
  - Production deployment and real personnel data. The human-decision comparison uses logged judgment on synthetic backlogs, not management of real people; a live pilot is a stretch goal only.

## Suggested Approach
This is a rough route, not a protocol. Adapt it as the system tells you what matters.
1. **Model the world first; the simulation is the foundation, and its realism is what makes any result believable.** Be explicit about what the sim does and does not claim to capture: a toy simulation that just mirrors your scheduler's own assumptions won't convince anyone. A reasonable starting shape:
   - A backlog spanning already-clean tickets to vague epics that must be decomposed. Each has a hidden true effort, dependencies, priority/value, and a deadline.
   - Workers, each a (speed, skill-by-area, reliability) profile. Given a ticket, a worker advances it over ticks; reliability governs blocking and the chance of returning buggy work that spawns a rework ticket; skill mismatch slows it and raises defect probability. Workers emit short standup updates. Cheap stochastic workers are fine to start; LLM-played workers are a stretch toward more realism.
   - A clock: discrete ticks (≈ days) over a sprint horizon.
   - Write down the assumptions the sim bakes in (effort distributions, how blocking/rework arise, whether reliability drifts within a sprint, how decomposition quality feeds downstream effort and defects). The goal is honesty about what's modeled, and ideally checking that a headline conclusion isn't just an artifact of one parameter setting: vary a couple of the load-bearing ones and see if the story holds. Treat that as good practice, not a gate.
2. **Build the tool API the manager acts through.** A thin interface: `list_backlog`, `create_ticket / split_ticket(acceptance_criteria…)`, `set_priority/estimate`, `assign(ticket, worker)`, `read_board / read_updates`, `flag_blocker / link_dependency`, `post_report`. Back it with the simulator now and with Jira/GitHub later, behind the same interface. This is the orchestrator–worker pattern: the manager plans and delegates and never does the work directly.
3. **Write the manager-agent loop, the heart of the system.** Each tick: ingest board + updates, decide actions (decompose, (re)prioritize, (re)assign, unblock), and at sprint cadence emit a status report. The aim is a manager that produces genuinely useful planning, assignment, and reporting: output you'd actually want to read. Keep memory and state explicit (a running plan plus per-worker load) to avoid the context-rot and lost-in-the-middle failures common in long-horizon multi-agent runs. Start with a single manager agent; add sub-agents only if a real failure demands it.
4. **Stand up simple baselines for comparison.** Run the manager against (a) random assignment, (b) round-robin, and (c) a reasonable heuristic scheduler (priority-first + skill-match + least-loaded). Give the heuristic a fair, honest shot (a deliberately weak baseline makes any win meaningless) but don't sink the project into baseline-tuning. Run on shared seeds and look at the spread across runs, not a single lucky one.
5. **Aim the comparison at where the agent should actually win.** A decent scheduler will likely match the agent on throughput and cycle time when the backlog is already clean, because that's a solvable optimization problem. The agent's plausible edge is upstream: turning vague, unstructured backlog items into well-formed, assignable tickets, and reacting to ambiguity a scheduler can't parse. So lean on the under-specified backlog, and make sure decomposition quality actually moves downstream outcomes in the sim. If it doesn't, the comparison can't see the thing that matters, so adjust the sim. Showing that better tickets lead to less rework or more on-time work is the most convincing form of the result.
6. **Collect human decisions and compare against them, the validity anchor.** Have a real manager (TA, mentor, you) work the same backlogs cold: how would they decompose this epic, who gets it, what's the priority order? Log those calls. Then compare the agent's choices to the human's: agreement rate on assignment/priority, and a side-by-side on how each decomposes the vague items. This is the load-bearing check: where the agent and human agree, the sim's "good" is corroborated by real judgment; where they diverge, dig in and decide who's right (sometimes the human, sometimes the agent, sometimes the sim is wrong). Without this, a sim win could just be the simulator congratulating its own assumptions. Keep the dataset small but real; this is core, not a stretch.
7. **Look at management quality from a few angles** to support the comparison, not to be the result; no single metric captures it (SPACE/Goodhart). Useful angles: throughput, cycle and lead time (mean and tail), on-time rate, rework/defect rate, load fairness, and decomposition quality (does each child ticket have a clear scope, acceptance criteria, and right size?). Score decomposition with a rubric and/or LLM-as-judge, and sanity-check a handful by hand so you trust the judge. Keep the axes separate; any composite score is itself a Goodhart target.
8. **Watch for gaming and bad calls.** Slip in an unmeasured-but-important factor (long-term tech-debt items, or fairness the throughput metric ignores) and see whether chasing the headline metric quietly degrades it. Log the confidently-wrong decisions (assigning to an overloaded or wrong-skill worker, ignoring a stated blocker) and write up what went wrong, leaning on the MAST taxonomy (spec ambiguity, inter-agent misalignment, verification gaps) as a vocabulary.

## Deliverables
- A working manager-agent system that does the upstream judgment (decomposing vague items into well-formed, well-assigned tickets) and coordinates a team end-to-end (decompose → prioritize → assign → unblock → report).
- A team + backlog simulator (configurable workers, dependencies, deadlines, stochastic blocking/rework, and a meaningful fraction of under-specified items) with seedable runs, plus notes on the assumptions it bakes in.
- **A human-decision dataset and comparison:** logged human-manager calls on the same backlogs, and an analysis of where the agent agrees, diverges, and which is right, the project's validity anchor.
- A tool API abstracting issue-tracker/git/chat, backed by the simulator and pluggable to a real tracker.
- Baselines (random, round-robin, a reasonable heuristic scheduler) on the same harness, to show what the easy part already solves.
- A supporting evaluation view comparing the agent to baselines and to the human decisions across the metric axes, a decomposition-quality rubric, and a check for metric-gaming.
- A short report: where LLM judgment beats rule-based coordination and where it only matches it, where it fails, and an honest line between automatable coordination and human-only management.

## Demo
One command runs a full sprint simulation. The manager agent gets a fresh backlog (including deliberately vague epics) and a team of simulated engineers, and live it decomposes the epics into well-formed tickets, assigns and prioritizes across workers, tracks progress and unblocks dependencies tick by tick, and emits a sprint status report. The view shows its decisions against the random / round-robin / heuristic baselines *and* against the logged human-manager calls on the same backlog, including a side-by-side on one vague epic: how the agent decomposed and assigned it versus how the human did, with outcomes (rework, on-time) for each. Plus one good and one bad managerial call walked through and whatever the gaming check surfaced.

## Success Criteria
- **Done:** simulator + tool API + manager loop + baselines run end-to-end on shared seeds, and a small human-decision dataset exists on the same backlogs. The agent does the upstream judgment (decomposes vague items into well-formed assigned tickets), and its calls can be compared against both the baselines and the human decisions.
- **Good:** the comparison earns a clear answer to the insight question. On clean backlogs the agent ties the heuristic (the easy part), but on vague items it does upstream judgment the heuristic can't, and that shows up as reduced downstream rework or better on-time rate. The human comparison corroborates this: where the agent agrees with the human manager the sim's "good" is validated, and the divergences are analyzed rather than waved away. Ideally one non-obvious finding surfaces (e.g. the agent and human diverge somewhere and the agent turns out right, or it games a metric until the multi-axis view exposes it), and the story isn't fragile to small changes in the sim's parameters.
- **Honest-eval bar:** the write-up is candid about where the agent fails and games (at least one concrete gaming example and a short taxonomy of its bad calls) and uses the human comparison to draw a clear line between coordination the agent can own and management that stays human. A leaderboard number alone isn't the point; whether LLM judgment actually beats rule-based coordination, and where, is.

## Tech & Prerequisites
- **Language:** Python (simulator, harness, dashboard).
- **Frameworks/tools:** an LLM API with tool/function calling (Anthropic/OpenAI); a simple agent loop (hand-rolled or a light framework); a dashboard (Streamlit or a static HTML report); optionally the GitHub Issues / Jira API for the real-tracker adapter (stretch).
- **Background:** comfort with LLM tool-calling and agent loops; basic discrete-event/stochastic simulation; experiment hygiene (seeds, replication, distributions over runs, holding conditions constant); familiarity with software-delivery metrics (DORA: lead time, change-failure rate; SPACE: no single number captures productivity).

## Phases
- **Explore:** read the starter refs; enumerate the concrete EM coordination functions and draw the automatable/human line; spec the worker model, backlog schema, and tool API; note the sim's assumptions; spike one sprint with a trivial heuristic manager.
- **Build:** implement the simulator, the tool API, the simple baselines, then the manager-agent loop; log its decisions.
- **Evaluate:** run all strategies across several seeds; collect the human-decision dataset and compare the agent against it; build the view; score decomposition quality and tie it to outcomes; check for gaming; write up the failures and the divergences.
- **Polish:** clean for one-command repro; write the report and the honest "automatable vs. not" section; prep the demo.

## Stretch Goals
- **LLM-played workers** instead of stochastic ones (more realistic standups, ambiguity, push-back), closer to managing agent-augmented engineers.
- **Real-tracker pilot:** point the same tool API at GitHub Issues/Jira and have the manager coordinate a couple of real coding-agent "engineers" on toy tickets in a sandbox repo. Low-stakes, live.
- **Scale up the human-decision dataset:** more managers, more backlogs, blind A/B on whose decompositions read better, sharpening the validity anchor that's already core to the project.
- **Human-in-the-loop mode:** require approval for high-impact actions and measure trust/override rate, which speaks directly to the accountability concern.

## Open Questions / Assumptions
- **Circularity of the simulator, the main risk.** A sim can reward exactly the decisions its designer baked in as "good," making the agent look smart for agreeing with its assumptions. The human-decision comparison is the core guard against this, not a stretch; the open question is how big and how diverse that human dataset needs to be before the agreement (and the divergences) carry weight. Worth confirming with the reviewer how much human data they'd want before trusting the results.
- **What counts as "management quality."** The metrics support the human and baseline comparisons rather than standing in for the answer, since no single number is defensible (SPACE/Goodhart). Which axes matter most is worth a conversation with the reviewer.
- **Decomposition-quality scoring** (rubric and/or LLM-as-judge) is subjective and the judge can itself drift. A written rubric and a few hand-checks should keep it honest; the more convincing signal is that better tickets actually change outcomes in the sim.
- **The agent's real edge is upstream** (handling vague, unstructured backlog items and turning them into well-formed assigned work) so it's most worth showing that edge as reduced downstream rework, not just a separate rubric score.
- **Over-automation and accountability are real.** A manager agent making confident bad calls, or gaming its own metrics, could do harm in a real org. Any non-simulated use should keep a human in the loop, and the human-in-the-loop stretch is the responsible path to a live pilot. Who's accountable when the agent mis-assigns or mis-reports is a governance question this project names rather than solves.
- Assumption: the manager acts only through a tool API (tracker/git/chat) and never edits code, keeping it an orchestrator rather than a coding agent.
